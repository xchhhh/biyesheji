**7张核心业务表**

```
┌────────────────────────────────────────────────────────┐
│ Table Schema Design                                    │
├────────────────────────────────────────────────────────┤
│ 1. users (用户表)                                     │
│    - 包含：学号、姓名、微信openid、信用积分、禁用状态│
│    - 索引：student_id, wechat_openid, credit_score   │
│                                                        │
│ 2. reading_rooms (阅览室表)                           │
│    - 包含：编码、名称、座位数、楼层、开放时间        │
│    - 索引：room_code, is_active                      │
│                                                        │
│ 3. seats (座位表)                                    │
│    - 包含：座位编号、行列、坐标、座位类型            │
│    - 索引：room_id, seat_type                        │
│    - 外键：room_id → reading_rooms                   │
│                                                        │
│ 4. reservations (预约记录表) ★重点表                 │
│    - 包含：用户、座位、日期、时间、状态、签到时间    │
│    - 索引：user_id, seat_id, reserve_date等6个      │
│    - 外键：user_id, seat_id, room_id               │
│    - 支持分区：按 YEAR_MONTH(reserve_date)分区      │
│                                                        │
│ 5. credit_logs (信用积分流水表)                      │
│    - 包含：用户、操作类型、积分变化、原因、余额      │
│    - 索引：user_id, created_at, operation_type      │
│    - 支持分区：按月分区便于清理旧数据              │
│                                                        │
│ 6. seat_status (座位实时状态表)                      │
│    - 包含：座位状态、预约ID、状态变更时间            │
│    - 索引：room_id, current_status                  │
│    - 一对一外键：seat_id → seats                    │
│                                                        │
│ 7. dashboard_stats (数据看板统计表)                  │
│    - 包含：统计类型、占用率、高峰时段、违规数等      │
│    - 索引：stat_date, room_id                       │
│                                                        │
└────────────────────────────────────────────────────────┘
```

**高级特性**

- ✅ **复合索引**: `uk_user_date_status` - 快速查询用户某日的预约状态
- ✅ **外键约束**: 保证数据一致性，级联删除
- ✅ **表分区**: 预约表和积分表支持按时间分区，降低表大小
- ✅ **视图**: 3个业务视图简化复杂查询
- ✅ **存储过程**: 自动化信用积分扣分流程
- ✅ **触发器**: 座位状态变更自动更新时间戳

### 第二部分：Redis缓存设计

**分层缓存架构**

```
应用请求
    ↓
[第1层] LocalCache (应用内存)
    ├─ 配置类数据
    ├─ TTL: 5-10分钟
    └─ 命中率最高
    ↓ miss
[第2层] Redis (分布式缓存) ★核心
    ├─ 座位状态 (24h)
    ├─ 用户会话 (24h)
    ├─ 热力图数据 (30m)
    ├─ 信用积分 (1h)
    ├─ 限流计数器 (24h)
    └─ 预约队列 (5m)
    ↓ miss
[第3层] MySQL (持久化)
    ├─ 完整数据存储
    ├─ 事务支持
    └─ 备份恢复
```

**缓存键设计规范**

```
模式: {module}:{sub_module}:{object_type}:{identifier}

示例:
  seat:status:1:101           # 1号阅览室的101座位状态
  user:credit_score:1001      # 用户1001的信用积分
  lock:reserve:1:101:2024-03-20:08:00
  queue:reserve:1:101:2024-03-20:08:00
  rate_limit:reserve:1001:2024-03-20
  room:heatmap:hourly:1:2024-03-20:08
  session:abc123def
```

**缓存容量预估**

```
总计: ~20MB内存用量（对于单实例Redis足够）

细分:
├─ 座位状态: 2000座位 × 500B = 1MB
├─ 用户会话: 1000在线用户 × 1KB = 1MB
├─ 热力图: 8时段 × 2000座位 × 100B = 1.6MB
├─ 用户预约: 10000用户 × 10条 × 100B = 10MB
└─ 其他缓存: 5MB
```

### 第三部分：高并发处理方案

**核心问题**: 早上8点，1000+学生同时抢座

**解决方案**

```
时间轴:
├─ T0: 用户点击预约按钮
├─ T+1ms: 检查限流 (RPS限制)
├─ T+5ms: 尝试获取分布式锁（30秒超时）
│   ├─ 成功: 继续处理
│   └─ 失败: 加入等待队列 (List)
├─ T+10ms: 最后一次检查座位状态 (Redis HGETALL)
├─ T+15ms: 原子性执行预约 (Lua脚本)
│   ```lua
│   if redis.call('HGET', seat_key, 'status') == 'free' then
│       redis.call('HSET', seat_key, 'status', 'reserved', ...)
│       redis.call('EXPIRE', seat_key, 86400)
│       return 1  # 成功
│   else
│       return 0  # 失败
│   end
│   ```
├─ T+20ms: 更新用户预约列表
├─ T+25ms: 异步将预约写入数据库队列
├─ T+30ms: 释放分布式锁
├─ T+35ms: 处理等待队列的下一个请求
└─ T+500ms: 客户端收到响应
```

**预期性能指标**

```
场景: 1000并发用户同时预约

┌─────────────────────────────────────┐
│ 性能指标          │ 期望值 | 达成方式 │
├─────────────────────────────────────┤
│ 平均响应时间       │ <500ms │ 异步DB   │
│ 预约成功率         │ >95%   │ 分布式锁 │
│ 座位重复预约率     │ 0%     │ Lua脚本  │
│ P99延迟            │ <2s    │ 限流器   │
│ 系统可用性(SLA)    │ 99.9%  │ 主从复制 │
└─────────────────────────────────────┘
```

## 💡 核心亮点

### 1. 高可用设计

- ✅ **Redis主从复制** - 自动故障转移
- ✅ **MySQL读写分离** - 读操作分散到从库
- ✅ **分布式锁** - 防止重复操作
- ✅ **幂等性设计** - 重试安全

### 2. 高性能设计

- ✅ **多层缓存** - 内存缓存 → Redis → MySQL
- ✅ **异步处理** - 数据库写操作异步执行
- ✅ **预约队列** - 平滑处理突发流量
- ✅ **Lua脚本** - 原子性操作，避免竞态条件

### 3. 高扩展设计

- ✅ **水平扩展** - 应用层可任意增加实例
- ✅ **表分区** - MySQL表按时间分区，支持清理旧数据
- ✅ **阅览室隔离** - 支持多阅览室并行处理
- ✅ **业务解耦** - 服务层和数据层分离

### 4. 安全设计

- ✅ **微信授权** - OpenID验证身份
- ✅ **JWT令牌** - 无状态认证
- ✅ **参数验证** - 防止SQL注入、XSS
- ✅ **频率限制** - DDoS防护
- ✅ **数据加密** - 传输层SSL，敏感字段加密存储

### 5. 可观测设计

- ✅ **结构化日志** - ELK Stack整合
- ✅ **性能监控** - Prometheus + Grafana
- ✅ **链路追踪** - Jaeger追踪请求
- ✅ **告警机制** - AlertManager自动告警
